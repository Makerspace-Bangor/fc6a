\begin{abstract}
Programmable Logic Controllers (PLCs) are industry-standard devices designed primarily for controlling high-voltage and industrial systems. Think of them as proprietary, industrial-grade Arduinos. They are not general-purpose computers, and often rely on obscure or archaic programming methods. First introduced in 1969, many PLCs still implement variations of Relay Control Logic (RCL), a system of representing electrical control diagrams that dates back over 160 years.

PLCs are complex in their simplicity. While their core concepts remain consistent, OEMs implement a wide range of communication protocols to enable control and data exchange. Through investigation, clear similarities emerge across these protocols; however, a fully open standard that enables true interoperability between platforms does not yet exist.

This lack of interoperability reinforces vendor lock-in. Once a system is built around a specific manufacturer’s ecosystem, future expansion becomes constrained by compatibility requirements. Even when more suitable or advanced components become available, the cost and complexity of redesigning the system discourages change. In practice, this often limits integration not only across different manufacturers, but even across product lines within the same OEM.

This project explores the development of interoperable control systems using Arduino and Arduino-compatible devices to interface with and control PLCs. The goal is to investigate practical pathways toward more flexible, open, and adaptable industrial control architectures. 

\end{abstract}

\section*{Nomenclature}

Every OEM uses different terminology for their communication protocols. Terms like DiSM, MiSM, and various “Maintenance” labels often describe systems that are functionally similar, even when branded differently. This is confusing. To avoid that, this document will refer to all such systems as \textbf{Maintenance Protocol}.

\textbf{Registers:} While micro-controllers traditionally use registers, this project targets interoperability across both computer and micro-controller platforms. For clarity and consistency, the terms \textbf{Registers} and \textbf{Addresses} may be used interchangeably where appropriate.

\textbf{Endian:} Refers to the order in which bytes are stored in memory. PLC OEMs may describe this using terms such as “byte packing,” “data format,” or “32-bit storage settings.” In this document, all such concepts of byte ordering and data layout will be referred to as \textbf{Endian}.

\textbf{Register Map:} Where specific data is stored. These mappings change from brand to brand. We are implementing an IDEC 
implementation of memory mappings from D0009 to D9999 and M0000 to M9999. The mappings change depending on system in use. I have a lot of PLCs, but use IDEC most often.

\section*{How The protocol works:}
The protocol request pattern is :  \\
ENQ = ACK/NAK    \\
DEVICE = FF/1:N  \#BYTE \\
CONTINUATION =   1: wait for more instructions, or 0: don't.\\
TYPE = Read / Write / Clear\\
Datatype: bits or registers basically\\
value=bytes varies based on datatype.\\
checksum=BCC, an XOR operation on the sum of the data being requested. \\
terminator = \verb|\0h| \\

You PLC is able to operate so fast, because it is just looping through memory as fast as it can. When it gets an instruction in a memory address it sends that instruction. Without delving into how PLC compiled code operates, too much, everything is a register to the PLC at run time. when the PLC has to execute an instruction, it is sending a frame following this pattern. When its communicating with internal devices, this pattern. When communicating with external devices, this framing patter is what it sends. Every time, for everything. The PLC then listens for a response, Acknowledge (AcK), and continues. The other resonse is NAk, No Acknowledge. The framing for NAk is only differs in the ENQ Byte.

\section*{Security:}
	PLCs are not quite computers, and practically speaking, have no security.  
If you are using PLCs, it is our advice and sincere hope that you do so behind a firewall or VPN.
Under no circumstance, give your PLC direct access to the internet. Yes, I know there are many features in integrations that point you to put these devices on the internet, and limited, basic example implementations of OEMs documenting doing just that, to send emails, connect to social media, run webservers, an other interesting IoT applications. Do it behind a firewall / vpn.
